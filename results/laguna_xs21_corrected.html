Appendix: Methodology & Testing Notes

A.1 — Why Published Tools Over Custom Scripts

The initial benchmark used a custom Python script that produced only 44-token outputs — too short to isolate decode from prefill and scheduling overhead. Corrected using vllm bench serve with 256-token forced outputs and --ignore-eos for clean steady-state measurement.

A.2 — GSM8K Strict vs Flexible Matching

GSM8K ships two scoring variants: strict-match requires the exact #### N format (0%), while flexible-extract regex-sweeps for any number matching the gold answer (35.94%). The model writes answers in prose rather than the canonical format — the 0% strict is a formatting artifact, not a capability gap.

A.3 — Few-Shot Attempt and Tokenizer Mismatch

An initial 5-shot GSM8K run produced near-random results due to a tokenizer conflict between lm-eval-harness and the model, causing prompt truncation in few-shot examples. Zero-shot configuration avoids this and produces clean results.
